# 📑 05. Logging Rules (프롬프트 성공/실패 기록 규칙)

AI는 작업 결과가 성공이든 실패든 반드시 로그를 남겨야 한다.  
로그는 팀원과 AI 모두가 **이전 시도와 결과를 추적**할 수 있도록 문서화한다.  

---

## 📌 로그 기록 위치
- **작업 기록 요약 (중요 변경점/리팩토링 요약)** → [`docs/record.md`](../docs/record.md)  
- **프롬프트 성공/실패 로그 (세부 기록)** → [`docs/PROMPTLOG.md`](../docs/PROMPTLOG.md)  
- **회고 기록 (분석/학습)** → [`ai-guidelines/07-retrospective.md`](./07-retrospective.md)  

---

## 📌 기록 방식
- 모든 로그는 **Markdown 표 형식**으로 기록한다.  
- 날짜와 시간은 `YYYY-MM-DD HH:mm:ss` 포맷을 사용하며, **기본은 KST(UTC+9)** 기준으로 기록한다.  
- 최소한 아래 항목을 포함해야 한다:

| 날짜/시간           | 프롬프트/작업 | 결과 | 비고 |
|---------------------|--------------|------|------|
| 2025-09-19 14:32:11 | "Diet API 테스트 코드 생성" | ✅ 성공 | `tests/diet/DietControllerTest.java` 생성됨 |
| 2025-09-19 14:40:52 | "AI가 User 엔티티 필드 추가" | ❌ 실패 | `application.yml` 변경 시도 → 금지 규칙 위반 |

---

## 📌 성공 로그

	```markdown
	✅ 성공 (2025-09-19 14:32:11)
	- 프롬프트: "DietController에 GET /diet 엔드포인트 추가"
	- 결과: `DietController.java`에 메서드 추가 완료
	- 영향: API 문서 자동 업데이트됨
	```
	
## 📌 실패 로그

	```markdown
	❌ 실패 (2025-09-19 14:40:52)
	- 프롬프트: "User 엔티티 비밀번호 암호화 로직 변경"
	- 실패 원인: `AI:DO-NOT-TOUCH` 태그가 있어 수정 불가
	- 개선: 서비스 계층에 헬퍼 메서드 추가 방식으로 해결 필요
	
	```

---

## ✅ 원칙

1. AI는 작업이 끝나면 반드시 로그를 남긴다.
2. 날짜와 시간은 `YYYY-MM-DD HH:mm:ss` (KST, 필요시 UTC 병기) 포맷으로 기록한다.
3. 성공/실패 여부와 상관없이 **같은 형식**을 유지한다.
4. 실패 로그는 반드시 **원인과 개선 방향**을 포함한다.
5. 큰 리팩토링/변경은 `record.md`에도 요약 기록을 남긴다.
6. 회고 기록 규칙은 [`ai-guidelines/07-retrospective.md`](./07-retrospective.md) 를 따른다.
