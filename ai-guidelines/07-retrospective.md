# 🪞 07. Retrospective Rules (회고 작성 규칙)

회고는 AI 로그와 작업 기록을 기반으로, 팀/리뷰어가 **개선 포인트를 정리하고 다음 사이클에 반영**하기 위한 문서이다.  

---

## 📌 회고 기록 위치
- 회고 문서는 [`docs/retrospectives/`](../docs/retrospectives/) 디렉토리에 작성한다.  
- 파일명 규칙: `retrospective-YYYY-MM.md` (예: `retrospective-2025-09.md`)  

---

## 📌 작성 주기
- **1~2주 단위**로 회고를 작성한다.  
- 스프린트/릴리즈 주기에 맞춰 유연하게 조정 가능하다.  

---

## 📌 회고 템플릿 (KPT 방식)
회고는 **Keep / Problem / Try** 3가지 항목으로 작성한다.

	```markdown
	# 📆 회고: 2025-09-19 ~ 2025-10-02
	
	## ✅ Keep (유지할 점)
	- AI가 주석 태그(`AI:CHANGE-ZONE`)를 잘 준수함
	- 코드 출력 시 설명 주석을 한글로 넣는 규칙이 잘 지켜짐
	
	## ⚠️ Problem (문제점)
	- 백엔드 테스트 코드에서 mocking 범위가 불명확함
	- 로그 기록 시 UTC 변환이 누락됨
	
	## 🚀 Try (개선할 시도)
	- `03-output-format.md`에 테스트 mocking 예시 추가
	- 로깅 유틸에 KST → UTC 자동 변환 기능 반영
	```

## 📌 분석 관점

- **성공 패턴**: 어떤 규칙/프롬프트가 잘 동작했는가?
- **실패 원인**: 반복된 실패나 금지 규칙 위반이 있었는가?
- **개선 시도**: 규칙 문서 업데이트, MCP 워크플로우 개선, 코드 템플릿 보강 등

---

## ✅ 원칙

1. 회고는 반드시 **팀/리뷰어 주도**로 작성한다.
2. AI 로그(`PROMPTLOG.md`)를 근거로 분석 내용을 작성한다.
3. Keep/Problem/Try 형식을 기본으로 하되, 필요 시 다른 프레임워크(4L, 5 Whys 등)도 활용 가능하다.
4. 회고 결과는 다음 사이클의 규칙 개정 및 AI 개선에 반영한다.
