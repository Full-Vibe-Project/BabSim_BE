# ✅ 제품 요구사항 정의서 (PRD) - 헬스케어 식단 관리 서비스

> 이 문서는 'AI 기반 헬스케어 식단 관리 서비스'의 제품 요구사항을 정의합니다. 모든 팀원(AI 포함)은 이 문서를 기준으로 기능 개발, 테스트, 문서화를 진행합니다.

**문서 버전: 1.0**
**최종 수정일: 2024-07-25**

---

## 1. 개요 (Overview)

### 1.1. 프로젝트명
- **헬스케어 식단 관리 서비스** (가칭: 밥심)

### 1.2. 목표
- AI 기술을 활용하여 식단 기록을 자동화하고, 개인의 건강 상태와 영양 섭취 패턴을 분석하여 맞춤형 식단과 건강 가이드를 제공함으로써, 사용자의 건강 관리 효율을 극대화한다.

### 1.3. 타겟 사용자
- **주요 타겟**: 건강한 식습관을 형성하고자 하는 20-40대 일반인
- **확장 타겟**: 당뇨, 고혈압 등 만성 질환 관리가 필요한 사용자

### 1.4. 해결하고자 하는 문제
- **수동 기록의 번거로움**: 기존 식단 기록 방식은 귀찮고 지속하기 어려워 건강 관리 습관화에 실패하는 경우가 많다.
- **단순 계산의 한계**: 대부분의 앱이 칼로리 계산에만 집중하여, 개인의 질병이나 특성을 고려한 맞춤형 관리에 한계가 있다.
- **데이터 활용 부족**: 축적된 건강 데이터를 장기적인 관점에서 분석하여 미래의 건강 리스크를 예측하고 예방하는 기능이 부재하다.

## 2. 주요 기능 (Core Features)

| 기능 ID | 기능명 | 핵심 설명 |
|---|---|---|
| **FE-01** | **사용자 관리** | JWT 기반의 안전한 회원가입 및 로그인 기능을 제공한다. |
| **FE-02** | **AI 식단 기록** | 음식 사진을 업로드하면 AI가 자동으로 음식을 인식하고 영양 정보를 추출하여 기록한다. |
| **FE-03** | **수동 식단 기록** | 사용자가 직접 음식 정보를 검색하고 섭취량을 입력하여 식단을 기록할 수 있다. |
| **FE-04** | **영양 리포트** | 일/주/월 단위로 섭취한 영양 성분을 분석하고 시각화된 리포트를 제공한다. |
| **FE-05** | **AI 맞춤 추천** | 사용자의 건강 프로필과 식단 데이터를 기반으로 부족/과잉 영양소를 분석하여 맞춤 메뉴를 추천한다. |
| **FE-06** | **Notion 통합** | 기록된 식단과 분석 리포트를 사용자의 Notion 페이지에 자동으로 동기화한다. |
| **FE-07** | **건강 위험 분석** | 장기적인 식단 데이터를 기반으로 비만, 당뇨 등 잠재적인 건강 리스크를 예측하고 알림을 제공한다. (Advanced) |

## 3. 기능 명세 (Functional Specifications)

### 3.1. 사용자 관리 (Auth)
- **(FE-01.1) 회원가입**
  - **설명**: 사용자는 이메일, 비밀번호, 이름 등 기본 정보를 입력하여 서비스에 가입할 수 있다.
  - **요구사항**:
    - 비밀번호는 안전하게 해싱하여 저장해야 한다. (e.g., BCrypt)
    - 이메일은 중복될 수 없다.
  - **API**: `POST /auth/register`
- **(FE-01.2) 로그인**
  - **설명**: 가입된 사용자는 이메일과 비밀번호로 로그인하고, 서비스 접근을 위한 JWT를 발급받는다.
  - **요구사항**:
    - 로그인 성공 시, Access Token을 발급한다.
    - 모든 인가된 API 요청은 `Authorization: Bearer <token>` 헤더를 포함해야 한다.
  - **API**: `POST /auth/login`

### 3.2. 식단 관리 (Diet)
- **(FE-02.1) AI 기반 식단 기록**
  - **설명**: 사용자가 음식 사진을 업로드하면, AI 모델이 이미지를 분석하여 음식 종류, 칼로리, 핵심 영양소(탄수화물, 단백질, 지방 등)를 자동으로 인식하고 결과를 반환한다.
  - **요구사항**:
    - 이미지 파일은 S3와 같은 외부 스토리지에 저장하고, URL 형태로 관리한다.
    - AI 분석 결과는 사용자가 수정할 수 있도록 편집 기능을 제공해야 한다.
  - **API**: `POST /diet/ai-entry`
- **(FE-03.1) 수동 식단 기록**
  - **설명**: 사용자가 식약처 음식 DB와 연동된 검색 기능을 통해 음식을 찾고, 섭취량을 입력하여 식단을 기록한다.
  - **요구사항**:
    - 식사 종류(아침, 점심, 저녁, 간식)를 구분하여 기록할 수 있어야 한다.
    - 기록 날짜와 시간을 지정할 수 있어야 한다.
  - **API**: `POST /diet/entry`
- **(FE-03.2) 일일 식단 조회**
  - **설명**: 특정 날짜에 기록된 모든 식단 내역을 조회할 수 있다.
  - **API**: `GET /diet/daily/:date`

### 3.3. 건강 데이터 (Health)
- **(FE-04.1) 주간/월간 데이터 요약**
  - **설명**: 지정된 기간(주/월) 동안의 평균 칼로리, 영양소 섭취량 등을 요약하여 제공한다.
  - **요구사항**:
    - AI가 "탄수화물 섭취가 평균보다 높습니다" 와 같은 간단한 텍스트 요약을 함께 제공한다.
  - **API**: `GET /health/summary/:period`
- **(FE-07.1) AI 기반 건강 위험 분석**
  - **설명**: 축적된 식단 데이터를 바탕으로 특정 건강 문제(예: 나트륨 과다로 인한 고혈압 위험)에 대한 리스크를 분석하여 사용자에게 알려준다.
  - **API**: `GET /health/risk-analysis`

### 3.4. AI 추천 (Recommendation)
- **(FE-05.1) 맞춤 메뉴 추천**
  - **설명**: 사용자의 건강 프로필(질병, 목표 등)과 최근 영양 섭취 상태를 종합하여, 현재 시점에 가장 적합한 메뉴를 추천한다.
  - **요구사항**:
    - 당뇨 환자에게는 고당 지수 음식을 제외하고 추천하는 등 개인화 로직이 포함되어야 한다.
  - **API**: `GET /recommendation/menu`

## 4. 비기능적 요구사항 (Non-Functional Requirements)

- **보안 (Security)**
  - 모든 사용자 비밀번호는 단방향 암호화하여 저장한다.
  - API 서버와 클라이언트 간의 모든 통신은 TLS/SSL을 통해 암호화한다.
  - 인증/인가가 필요한 모든 API는 JWT를 통해 보호된다.
- **성능 (Performance)**
  - 모든 API의 평균 응답 시간은 500ms 이하를 목표로 한다.
  - AI 분석과 같이 오래 걸리는 작업은 비동기로 처리하여 사용자 경험을 저해하지 않도록 한다.
- **데이터 관리 (Data Management)**
  - 사용자의 개인정보 및 건강 데이터는 관련 법규를 준수하여 안전하게 관리한다.
  - 식약처 제공 음식 영양성분 DB를 초기 데이터로 구축하며, 주기적으로 업데이트한다.
- **확장성 (Scalability)**
  - 시스템은 Feature-first 아키텍처를 기반으로 설계되어, 새로운 기능(도메인)을 독립적으로 추가하기 용이해야 한다.
  - AI 서비스, 외부 API, 파일 저장소 등은 인터페이스 기반으로 설계하여 다른 구현체로 쉽게 교체할 수 있어야 한다.
